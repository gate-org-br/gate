@search

The @search trigger enhances user interaction by providing a dynamic search dialog. Accepting a filter as parameter, it displays a dialog featuring a search input field for users to choose an element from a JSON array fetched from the server. The search input triggers requests to the server, either as get requests with search criteria in the query string (if a filter attribute is supplied) or as post requests with user input as the body (if the filter is omitted). The selected object is then dispatched to the next trigger on the pipeline. To optimize performance, a debounce of 300ms is applied, preventing unnecessary server requests. Additionally, when triggered by text input, the dialog opens with the input value pre-applied to the search field. If the server returns a single value, it is automatically selected. This functionality streamlines the user experience while ensuring efficiency in handling search queries.

Example 1 –

When the action returns an JSON array of objects they are presented as cards on a list for user selection. The first attribute of each object is not displayed as it contains the value to be returned. Upon user selection, the label input receives the first attribute and the value input receives the second one.

Example 2 –

If the result consists of a JSON array of objects with only label and value attributes, the labels will be displayed on a list for user selection. Upon user selection, the label input and the value input receives the label attribute and the value attribute respectively.

Example 3 –

If the result consists of a JSON array of objects with only label, value and properties attributes, the properties will be displayed as cards on a list for user selection. Upon user selection, the label input and the value input receives the label attribute and the value attribute respectively.
