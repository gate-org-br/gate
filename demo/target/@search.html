@search

The @search target enhances user interaction by providing a dynamic search dialog. Accepting a filter as parameter, it displays a dialog featuring a search input field for users to choose an element from a JSON array fetched from the server. The search input triggers requests to the server, either as get requests with search criteria in the query string (if a filter attribute is supplied) or as post requests with user input as the body (if the filter is omitted). The selected object is then dispatched to the next target on the pipeline. To optimize performance, a debounce of 300ms is applied, preventing unnecessary server requests. Additionally, when triggered by text input, the dialog opens with the input value pre-applied to the search field. If the server returns a single value, it is automatically selected. This functionality streamlines the user experience while ensuring efficiency in handling search queries.

Example 1 –

When the action returns a JSON array of objects, the @search target showcases them as cards within a list for user selection. Any attributes beginning with the _ character are omitted from display. Upon user selection, the chosen object is directed to the @fill target. When invoked without parameters, the @fill target populates the value of the first hidden input and the first text input, which are siblings of the trigger element, with the first and second properties of the selected object, respectively.

Example 2 –

When the action returns a JSON array of arrays, the @search target showcases them on a grid for user selection. Any columns with headers beginning with the _ character are omitted from display. Upon user selection, the chosen array is directed to the @fill target. When invoked without parameters, the @fill target populates the value of the first hidden input and the first text input, which are siblings of the trigger element, with the first and second columns of the selected array, respectively.
